# HydroMind.AI Website Audit Log — 2026-07-19

## Scope

- Live site: `https://www.hydromindai.com`
- Local deployment source: `Web/`
- Reviewed: public page reachability, response headers, Vercel API handlers, and authentication/authorization paths.

## Fixed

### HM-SEC-001 — Critical: KB administration authorization bypass

**Affected handlers:** `Web/api/kb-documents.js`, `Web/api/kb-upload.js`, `Web/api/kb-reembed.js`, and the deployed backend's `/api/kb/upload`, `/api/kb/documents`, `/api/kb/documents/:id/status`, and `/api/kb/documents/:id` routes.

**Cause:** The handlers accepted a caller-provided `Authorization: Email <address>` value (or `userEmail` in the request body). Any caller who knew a premium user's email could operate on the shared knowledge base; the handlers did not validate a session token or require administrator privileges.

**Impact:** Unauthorized list, upload, re-embedding, and deletion of KB documents/chunks. Because the handlers use the Supabase service-role key, successful abuse bypasses normal row-level protections.

**Remediation applied:** Added a shared server-side guard that validates a Supabase bearer token with `auth.getUser()` and requires `users.is_admin = true` before allowing KB administration. The Vercel handlers now return `403` when the caller is not an authenticated administrator, and CORS is restricted to the production web origin. The Render backend now looks up the live `is_admin` field after authenticating each KB-management request and returns `403` for non-admin users.

**Verification required after deployment:** Sign in as an admin, obtain the normal session bearer token, and confirm upload/list/delete/re-embed still work. Confirm a premium non-admin and an anonymous request each receive `403`.

### HM-SEC-004 — Medium: stored DOM XSS in account navigation

**Affected file:** `js/auth-v2.js`

**Cause:** The navigation user chip inserted account name and plan values through `innerHTML`.

**Remediation applied:** Rebuilt the chip and sign-in error hint with DOM nodes and `textContent`, so dynamic values are rendered as text rather than executable markup.

### HM-SEO-001 — Low: sitemap validation failed

**Affected files:** `sitemap.xml`, `scripts/validate-static-site.mjs`

**Cause:** Sitemap modification dates were stale and the validator did not map the published circuit walkthrough page.

**Remediation applied:** Updated sitemap dates to the matching source files and added the published route to the validation map.

### HM-SEC-002 — High: unauthenticated AI and embedding proxies

**Affected handlers:** `api/chat.js`, `api/kb-search.js`

**Cause:** These unused Vercel routes accepted unauthenticated requests and spent Anthropic/OpenAI API credits with server-held keys.

**Remediation applied:** Retired both routes with an explicit `410 Gone` response. The reviewed frontend source uses the rate-limited Render backend endpoints instead.

## Open items

### HM-SEC-003 — Medium: feedback endpoint can be abused for spam

`api/feedback.js` is intentionally public and has no abuse control. Add a server-side rate limit and a bot challenge/verification appropriate to the deployment platform; CORS alone does not prevent direct scripted requests.

### HM-OPS-001 — Low: stale source artifacts increase review and deployment risk

The web directory contains numerous `.bak*`, `.orig`, and duplicate legacy files. They should be moved out of the deployable root or excluded from deployment after a confirmed backup, so old auth and UI code cannot be accidentally reactivated.

## Live-site observations

- HTTPS, HSTS, `nosniff`, strict referrer policy, and a CSP are present.
- The CSP allows inline scripts/styles because the site relies on them. Removing `unsafe-inline` requires moving inline code to external files and using nonce or hash policies; this was not changed in this targeted security remediation.
- The live page is reachable and the primary public routes load. The live pages show inconsistent KB counts (for example, 155 on home/KB pages and 86+ on the advisor), which is a product-content consistency issue to correct separately.
